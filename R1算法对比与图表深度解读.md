## 1. 新旧 R₁ 算法对比（带代码）

### 1.1 一句话区别

| | 旧算法（昨天之前） | 新算法（昨天 commit `10f499c` 之后） |
|---|---|---|
| 取 (P_ref, P_actual) 的时机 | **只取最后一轮** `round_logs{end}` | **累加所有轮次**的发电机出力 |
| φ 是什么 | "最后一轮幸存发电机的瞬时健康度" | "整个级联过程中，所有时刻所有发电机的加权平均健康度" |
| 方差大不大 | **大**（最后一轮可能只剩 1-2 台机） | **小**（被全程的几十次采样平均了） |
| 排序稳不稳 | 不稳，medium / heavy 经常颠倒 | 稳，5 个场景严格分层 |

### 1.2 R₁ 的总公式（新旧都一样）

```
R₁ = (存活负荷 × φ) / 初始总负荷
       │           │
       │           └── delay_penalty，通信时延造成的"出力打折"系数
       └── surviving_load，纯结构性损失（节点死了拉走的负荷）
```

代码在 `computeDelayAdjustedR1.m`，第 22-24 行：

```matlab
delay_penalty = min(1, actual_generation / reference_generation);   % φ
delayed_served_load = surviving_load * delay_penalty;                % 存活负荷 × φ
R1_delay = delayed_served_load / initial_total_load;                 % 除以初始总负荷
```

> **这个函数本身没变**。变的是**喂给它的 `P_actual` 和 `P_ref` 是哪些数**。

### 1.3 旧算法（已废弃）—— 只看最后一轮快照

旧版本的代码逻辑（已被替换，等价代码片段）：

```matlab
% 旧逻辑：只取 round_logs{end}
last_rl = round_logs{end};
dil = last_rl.delay_injection_log;          % 最后一轮的 η 日志
P_ref_last = []; P_actual_last = [];
for gk = 1:numel(dil.eta)
    match = find(mpc.gen(:,1) == dil.gen_bus(gk), 1, 'first');
    if ~isempty(match) && abs(mpc.gen(match,2)) > eps
        P_ref_last(end+1)    = mpc.gen(match, 2);                    % 该机的额定出力
        P_actual_last(end+1) = mpc.gen(match, 2) * dil.eta(gk);      % 实际出力 = 额定 × η
    end
end
R1_mat(...) = computeDelayAdjustedR1(initial_power_load, failed_pn, P_actual_last, P_ref_last);
```

**问题在哪里？** 假设 39 节点电网级联完了只剩 2 台幸存发电机：

- A 试验：剩下的两台都靠近 CC（比如 G2、G3），路径短 → η ≈ 0.9 → φ ≈ 0.9 → R₁ 看起来很高
- B 试验：剩下的两台都在远端（比如 G9、G10），路径长 → η ≈ 0.4 → φ ≈ 0.4 → R₁ 看起来很低

A 和 B 可能是**完全相同的攻击在不同 BA 拓扑下的两次试验**，结构性损失（surviving_load）也差不多，但 R₁ 因为"最后一轮恰好剩谁"被随机决定，差出 50%。这就是箱体被撑得很宽、五个场景互相覆盖的根本原因。

> 大白话：旧算法就像考试只算"最后一道题"的分。同一个学生不同卷子最后一道题难度不同，分数完全不可比。

### 1.4 新算法 —— 全程累计加权（φ_traj）

代码在 `bet_homo_gudingCC_myself_bet_8.m:193-303`：

```matlab
% 第 199-200 行：初始化全程累计向量
P_ref_traj = [];
P_actual_traj = [];

% 第 219-248 行：逐轮循环时，每一轮都把这一轮的发电机加进累计向量
for roundIdx = 1:num_rounds
    rl = round_logs{roundIdx};
    if isfield(rl, 'delay_injection_log') && ~isempty(rl.delay_injection_log.eta)
        dil_round = rl.delay_injection_log;
        P_ref_round = [];
        P_actual_round = [];
        for gk_round = 1:numel(dil_round.eta)
            match_round = find(mpc.gen(:,1) == dil_round.gen_bus(gk_round), 1, 'first');
            if ~isempty(match_round) && abs(mpc.gen(match_round, 2)) > eps
                pg_ref_round = mpc.gen(match_round, 2);
                P_ref_round(end+1, 1)    = pg_ref_round;                          % 该轮该机的额定 P
                P_actual_round(end+1, 1) = pg_ref_round * dil_round.eta(gk_round);% 该轮该机的实际 P
            end
        end
        if ~isempty(P_ref_round) && sum(P_ref_round) > 0
            % ★ 关键：把这一轮的 (P_ref, P_actual) 追加到全程累计向量
            P_ref_traj    = [P_ref_traj;    P_ref_round];
            P_actual_traj = [P_actual_traj; P_actual_round];
        end
    end
end

% 第 298-303 行：循环结束后，用全程累计的两个长向量一次性算 R₁
if ~isempty(P_ref_traj) && sum(P_ref_traj) > 0
    R1_mat(idxAlpha, trial, idxScenario) = computeDelayAdjustedR1( ...
        initial_power_load, failed_pn, P_actual_traj, P_ref_traj);
end
```

**φ_traj 的数学含义**：

$$
\varphi_{\text{traj}} \;=\; \frac{\sum_{r=1}^{R}\sum_{g \in \mathcal{G}_r} P_{\text{ref},g} \cdot \eta_{g,r}}{\sum_{r=1}^{R}\sum_{g \in \mathcal{G}_r} P_{\text{ref},g}}
$$

- 双重求和：外层是**所有级联轮次** r=1..R，内层是**该轮所有有 η 记录的发电机** g。
- 权重是 `P_ref`（额定出力大的机权重大，物理上合理：500 MW 的机 η=0.5 比 50 MW 的机 η=0.5 影响大十倍）。
- 分子是"实际能产出的总能量"，分母是"如果没时延能产出的总能量"。

> 大白话：新算法把整个级联从开始到结束**每一轮都打一次分**，最后取按发电机大小加权的平均分。哪怕最后一轮幸存的机器是好是坏都不会主导结果。
> 类比：新算法是"平时分 + 期末分加权平均"，旧算法是"只看期末"。

|---|---|---|

### 1.6 为什么旧算法当时是"合理选择"，现在又必须改？

旧算法源自一个**合理的工程直觉**："级联结束后系统才稳定，最后一轮的状态就是最终状态。" 在没有"全程跨轮跟踪"基础设施前，这是最简单的实现。

但是 `cascadeLogicdebug2gudingCC_bet_8.m` 后来加上了 `round_log` 机制（每一轮都把 `delay_injection_log` 存下来），就为"全程加权"打开了实现空间。一旦你需要把 5 个场景**精确分层**而不是"看个大概趋势"，全程加权就成了必须。

> 答：**两者都要看，但看的是不同东西**。

### 2.1 图 1b 的标题和坐标

| 坐标 | 含义 |

| 纵坐标 ΔR₁ | **同一个 trial** 在 no_delay 场景下的 R₁ 减去同一 trial 在该场景下的 R₁ |

### 2.2 "配对"两个字的关键含义

代码 `:489-501`：

```matlab
for ai = 1:numel(alpha_repr_idx)
    idxA = alpha_repr_idx(ai);
    r1_nd_trials = squeeze(R1_mat(idxA, :, nodelay_box_idx));   % no_delay 在这个α下500次的R1
    for idxS = 1:num_delay_scenarios
        r1_sc_trials = squeeze(R1_mat(idxA, :, idxS));          % 当前场景在这个α下500次的R1
        valid_mask = ~isnan(r1_nd_trials) & ~isnan(r1_sc_trials);
        delta_vals = r1_nd_trials(valid_mask) - r1_sc_trials(valid_mask);  % ★ 同 trial 配对作差
        bp_delta_data = [bp_delta_data; delta_vals(:)];
        ...
    end
end
```

**关键**：`r1_nd_trials(k) - r1_sc_trials(k)` 是**第 k 次试验**两个场景的差值。这两次试验跑的是**相同的 BA 拓扑、相同的初始攻击节点、相同的 RNG 种子**，只是时延配置不同。所以 `delta_vals(k)` 这一个数已经把"拓扑/RNG 噪声"消掉了，**剩下的就是时延强度的纯效应**。

### 2.3 纵坐标的值代表什么（盯中位数线）

每个箱子内有一条**红色横线（中位数）**，它的纵坐标值含义是：

> **"在这个 α、这个时延场景下，500 次试验里，时延比 no_delay 多吃掉的 R₁ 中位数。"**

举例（数字示意，以你的图为准）：
- α=0.5、heavy 箱中位数 = 0.42 → **意思是"heavy 比 no_delay 平均要多损失 42% 的初始负荷"**
- α=0.5、light 箱中位数 = 0.06 → light 只让你多损失 6%
- α=0.5、no_delay 箱中位数 = 0（自己减自己）

**判分标准**：
- 中位数 > 0 → 该场景**确实比 no_delay 差**（时延有害）。值越大伤害越大。
- 中位数 = 0 → 该场景和 no_delay 没差。
- 中位数 < 0 → 该场景**反而比 no_delay 好**（理论上不该出现，如果出现说明有 bug 或随机起伏）。

### 2.4 箱子长度（IQR）代表什么（盯箱体高度）

箱子上沿是 Q75，下沿是 Q25，箱长 = IQR = Q75 − Q25。它的含义是：

> **"在这个 α 下，时延伤害本身的不确定性大不大。"**

- 箱子**短** → 不同 BA 拓扑/初始攻击下，时延造成的额外损失基本一致 → 时延效应**可预测**。
- 箱子**长** → 时延的伤害高度依赖具体拓扑 → 时延效应**碰运气**。

经验观察：α 较小（系统在崩溃边缘）时配对差的箱子会长一点（杠杆区不稳定）；α 大（结构余量充足）时箱子短而稳定。

### 2.5 总结：两个维度怎么搭配看

| 看什么 | 回答什么问题 |
|---|---|
| **中位数纵坐标值**（绝对位置）| "时延平均**多伤害**了多少？" |
| **箱长**（IQR）| "这个伤害**稳不稳定**、能不能预测？" |
| **箱子之间的纵向间距** | "提升一档时延强度**追加**多少伤害？"（衡量边际效应）|
| **箱子是否横跨 y=0 线** | "是否存在'反而更好'的反直觉试验？"（异常诊断）|

> **一句话给你做汇报用**：图 1b 的纵坐标告诉我们"时延造成多少额外伤害"，箱子长度告诉我们"这个估计有多稳"。两个一起看，才能既知道**平均效应**又知道**置信程度**。

## 4. 动作实验重设计——让通信侧动作排第 1

### 4.1 为什么 A4/A5 现在永远第一？算法/物理两层原因

**物理层面**：时延公式

```matlab
% cascadeLogicdebug2gudingCC_bet_8.m:528-529
tau_m_g = delay_cfg.power.pb_to_noncc_measurement_delay_s + cyber_up_d;
tau_e_g = delay_cfg.power.noncc_to_pb_execution_delay_s   + cyber_down_d;
```

把这两个 τ 在 heavy 场景下**展开**：

| 分量 | 数值（heavy = scale 3.5）| 占 τ_m 比例 |
|---|---|---|
| `power.pb_to_noncc_measurement_delay_s` | 100 ms × 3.5 = **350 ms** | ≈ 99% |
| `cyber_up_d`（链路+端点+转发，最坏几跳）| ~3-5 ms | ≈ 1% |

τ_e 同理：420 ms vs ~3 ms。

→ A4/A5 把那 99% 的电力侧端口时延一刀砍到 20-30 ms，**剩下的几张牌（A1/A2/A3）全在那 1% 里折腾**，怎么打都赢不了。

**算法层面**：η = (1−k_m·τ_m)(1−k_e·τ_e) 是**乘性**的，所以哪个分量大，削减谁的边际收益就大。

### 4.2 你的目标拆解

> "我想让 A4 或 A5 排到**第 2 或第 4**，让通信侧动作能排第 1。"

要达到这个目标，**唯一可行的两条路**：

#### 路线 A：让"通信侧时延"在 τ 里也变得很大

让 cyber_up_d / cyber_down_d 在 heavy 场景下从 ~3 ms 涨到与 350 ms **同量级**（比如 200-500 ms），这样通信侧动作（升级带宽、缩短转发）才有"砍大刀"的空间。

#### 路线 B：让"通信侧改进动作"做得比 A4/A5 更激进

比如设计一个"骨干网光纤化 + 全网 SDN 调度"的复合动作，一次性把通信路径时延砍 90%，同时让 heavy 场景的通信侧时延本身就比较大。

**两条路本质上是同一回事**：先把通信侧时延"催大"成主导项，再用通信动作精准削减。

### 4.3 具体做法（建议——还没改代码，先和你确认）

#### Step 1：在 `createDelayConfig.m` 里把通信侧基准拉高

把通信参数从"几毫秒级"提升到"几十~百毫秒级"，例如：

| 参数 | 当前值 | 建议改成 | 理由 |
|---|---|---|---|
| `communication.default_link_rate_bps` | 10 Mbps | **1 Mbps** | 让 packet/rate 项变成几十 ms（1024×8/1e6 = 8 ms × scale 3.5 = 28 ms × 多跳）|
| `communication.default_distance_km` | 1 km | **50-200 km** | 模拟广域网，传播时延变成 0.25-1 ms × 多跳 × scale |
| `service.noncc.forward` | 6 ms | **30-50 ms** | 模拟 nonCC 节点处理慢（弱算力工业网关）|
| `service.cc.forward` | 3 ms | **15-25 ms** | 同理 |
| `service.noncc.tx`/`rx` | 12/9 ms | **30/20 ms** | 端点协议栈处理慢 |

这样 heavy 场景下 cyber_up_d/cyber_down_d 会被 scale 3.5 推到 **几百毫秒**，跟电力侧端口时延同一个量级。

> 物理意义：从"理想专网"模型改成"现实广域工业通信网"模型——**这其实更贴近 CPPS 真实场景**，并不是为了让结果好看而强行改。

#### Step 2：重新设计 5 个动作（建议方案，待讨论）

把现有 5 个动作改成"3 个通信 + 2 个电力"，并让通信动作的削减幅度变激进：

| 编号 | 名称 | 干什么 | 期望排名 |
|---|---|---|---|
| **A1（新）** | 骨干网光纤升级 | `default_link_rate_bps` 1 Mbps → 1 Gbps（×1000）+ `default_distance_km` /5（多跳光纤直连） | **第 1**（砍最大块）|
| **A2（新）** | SDN 路径优化 | `service.noncc.forward` heavy 值减 80% + `service.cc.forward` heavy 值减 70%（智能路由跳过弱节点）| **第 2**（砍多跳累计）|
| **A3（新）** | 端点协议栈加速 | `service.noncc.tx/rx` 和 `service.cc.tx/rx` heavy 值各减 70%（DPDK/RDMA） | **第 3** |
| **A4（保留）** | 测量接口加速 | 350 ms → 20 ms | **第 4**（仍有价值但不再独大）|
| **A5（保留）** | 执行接口加速 | 420 ms → 30 ms | **第 5**（同 A4） |

#### 6.1.3 颜色矩阵的算法

代码 `:1102-1114`：

```matlab
action_recovery_by_round_best  = NaN(plot_max_round, numA);
action_recovery_by_round_worst = NaN(plot_max_round, numA);
for r = 1:plot_max_round
    for a = 1:numA
        denom = mean_ts_R1(r, a, nodelay_idx) - mean_ts_R1(r, a, heavy_idx);
        if denom > 0.001
            action_recovery_by_round_best(r, a) = ...
                (mean_ts_R1_action(r, a, best_action_idx)  - mean_ts_R1(r, a, heavy_idx)) / denom * 100;
            action_recovery_by_round_worst(r, a) = ...
                (mean_ts_R1_action(r, a, worst_action_idx) - mean_ts_R1(r, a, heavy_idx)) / denom * 100;
        end
    end
end
```

这定义了二维矩阵：

$$
\text{Recovery}\%(r, \alpha) = \frac{\overline{R_1}^{\text{action}}(r,\alpha) - \overline{R_1}^{\text{heavy}}(r,\alpha)}{\overline{R_1}^{\text{no\_delay}}(r,\alpha) - \overline{R_1}^{\text{heavy}}(r,\alpha)} \times 100\%
$$

- 分子：动作把 heavy 提升了多少（与 heavy 的差）
- 分母：no_delay 比 heavy 高多少（"理论可恢复总量"）
- 比值：动作恢复了多少比例的时延伤害

`mean_ts_R1` 和 `mean_ts_R1_action` 都是**逐轮的 delay-adjusted R₁ 跨 trial 平均值**（即新算法下逐轮算的 R₁，再对 500 次 trial 取均值）。

#### 6.1.4 图中信息

1. **左子图整体亮度 ≥ 右子图**：在任意 (round, α)，最佳动作的恢复比例都不弱于最差动作。
2. **两子图共有的"亮区"**：通常出现在 Round 2-4 × 中高 α (0.4-0.7)，对应"杠杆区+反馈刚点火"。
3. **Round 1 整列偏暗**：第 1 轮还没注入时延，没有"伤害"可恢复，分子 ≈ 0。

#### 图里能看到什么？

1. 左图整体黄/红一片，右图整体偏暗 → **好动作不管哪儿都比差动作强**
2. 两张图都在"中间偏左、纵轴中段"那块最亮 → **早期的中等 α 是黄金窗口**
3. 第 1 列（Round 1）两张图都黑 → **第 1 轮所有场景都一样，没东西可救**
4. 第 7、8 列（晚期）颜色变淡或不稳 → **系统已经定型了，做什么都晚了**
5. 最下面一行（α=0）多半是空的或不稳 → **系统崩光了，分母接近 0，没意义**

#### 这些颜色是怎么算出来的？

每个格子的颜色 = **救回的伤害比例**：

```
（用动作之后的 R₁  −  完全不管的 heavy R₁）
─────────────────────────────────────────  × 100%
（理想 no_delay 的 R₁  −  完全不管的 heavy R₁）
```

- **分子** = "做了动作以后，比啥都不做提升了多少"
- **分母** = "no_delay 比 heavy 一共能高多少"（这是上限）
- **比值** = 救回了上限的百分之多少

代码里就是 `:1108-1111` 那两行除法。

#### 为什么会这样？

- **左图比右图亮**：左图选的是"砍 350 ms 端口时延"的 A4，右图选的是"换网线"的 A1。砍大头自然救得多。
- **Round 2-4 + 中高 α 最亮**：第 2 轮是反馈刚点火，这时给一脚刹车最有用；中高 α 是"将死未死"的临界点，η 改一点就能决定"过不过载"。
- **Round 1 黑**：因为级联代码里时延是在每轮**末尾**注入的，第 1 轮还没轮到注入，所有场景 η 都一样，差值 = 0，比值 = 0。
- **晚期暗**：反馈跑完了，能死的死了能断的断了，剩下的怎么救都差不多。
- **低 α 空白**：分母太小，怕除出来全是噪声，代码主动过滤掉了 (`if denom > 0.001`)。

#### 为啥要做这张图？

光看图 8（4 根柱子）有人会怀疑："你说 Round 2-3 是最佳时机，但万一只是某个特定 α 下凑巧呢？"

图 9 就是"补刀的证据"：把每个 (轮次, α) 全摊开来看，证明：
- 最佳动作**在所有位置都更好**（不是 cherry-pick）
- 绿色虚线（图 8 选的那几轮）**确实穿过了亮区**（不是瞎选）
- "什么时候干预什么 α 系统最值"是有规律的（不是噪声）

如果你要给老师/答辩讲：
> "图 9 是图 8 的展开，把 4 根柱子背后的 (轮次 × α) 二维分布全部摆出来。你能看到无论用什么动作，热度都集中在级联早期+中高 α，这就是我们建议的'黄金干预窗口'。"
